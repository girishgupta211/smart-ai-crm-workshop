Slide 1 / Foundations

How Claude, Cursor, and Codex work

AI coding tools are context engines plus action interfaces. They read your goal, inspect files, propose changes, run checks, and improve through feedback.

Prompt Project files Constraints Feedback loop
Input User goal, selected files, terminal output, docs, errors
Reason Infer intent, detect patterns, plan edits, choose next actions
Act Write code, explain tradeoffs, run tests, revise implementation

Key idea

Better context produces better output. AI is useful, but it still needs specs, tests, and review.

Slide 2 / Tool choices

Claude vs Cursor vs Codex

Same core idea, different surfaces for thinking, editing, and repo-level execution.

Claude

Best for reasoning before implementation.

  • PRDs and product thinking
  • Architecture tradeoffs
  • Code review and explanation
  • Prompt and workflow design

Cursor

Best inside the editor while writing code.

  • Inline edits
  • Codebase navigation
  • Fast refactors
  • Pair-programming loops

Codex

Best for repo-level tasks with terminal checks.

  • Multi-file changes
  • Scripts and tests
  • Debugging from logs
  • Implementation to validation

Slide 3 / Tool bridge

What is an MCP server?

An MCP server is the bridge between an AI chat and real tools. The chat decides what it needs, the MCP server safely exposes what it is allowed to use.

AI chat Claude, Cursor, or Codex receives your request and decides whether it needs external facts or actions. Example: "Generate a strategy for lead 2."
MCP server Defines available tools, validates inputs, runs controlled code or API calls, and returns structured results. It is the safety boundary between model reasoning and your systems.
Tools + data CRM data, files, GitHub, databases, APIs, docs, and internal business actions. The model does not access everything directly.
1. User asks Chat gets a goal.
2. Model plans It chooses a tool.
3. Server runs MCP executes safely.
4. Result returns Facts go back to AI.
5. AI answers Now grounded in data.

Slide 7 / System design

Complete app architecture

The product has four clear layers: frontend, backend API, database, and AI workflow.

Frontend React + Vite CRM dashboard: metrics, lead grid, details drawer, interaction timeline, AI strategy pane.
Backend API Express REST endpoints for leads, interactions, and strategy generation.
Database PostgreSQL tables: users, leads, interactions, and ai_strategies.
AI Layer Agent workflow plus MCP-style tools that fetch facts and save structured strategy output.

Request path

1. User action User selects a lead and clicks Generate Strategy.
2. API orchestration Backend starts the agent workflow and exposes controlled tool calls.
3. Data lookup Tools read exact metrics and latest interactions from the database.
4. UI refresh Saved strategy returns to the frontend and appears in the AI workspace.

Slide 8 / Reliable AI systems

MCP and agents: simple mental model

An agent is the planner. MCP tools are the approved actions it can take. The model should ask tools for facts before it answers.

1. Agent receives goal User says: generate a strategy for lead 2. The agent plans what information it needs first.
2. Agent calls tools get_lead_metrics gives exact numbers. get_lead_context gives latest notes.
3. Agent writes result After reading tool results, it calls update_lead_strategy to save a structured answer.

Why this matters

Without tools, the model may guess. With tools, it follows a predictable path: UI -> Agent -> get_lead_metrics -> get_lead_context -> update_lead_strategy -> UI. Guardrail: no notes means Insufficient context to form strategy.
